OVERVIEW

Build the order detail view (single order with full info + status timeline),

This is what the merchant sees when clicking an order from the list.


BACKEND TASKS

  [ ] 1. Create detail/update/delete schemas
      File: backend/app/schemas/orders.py (add to existing)

      OrderDetailResponse:
        - All fields from OrderResponse

        - store:           Optional[StoreInfo]   (name, address, phone)

      StatusHistoryItem:

      StatusUpdateRequest:
        - status: str   (assigned | picked_up | in_transit | delivered | cancelled)
        - note:   Optional[str]

  [ ] 2. Create detail/update/status services
      File: backend/app/services/order_service.py (add to existing)

      async def get_order_detail(db, merchant_id, order_id) -> dict:
        - Fetch order by UUID (not order_id string) 
        - Verify order belongs to merchant
        - Join with stores table for store info
        - Fetch status_history ordered by changed_at ASC
        - Return combined data

      async def update_order(db, merchant_id, order_id, data) -> dict:
        - Verify order belongs to merchant
        - Verify order status is 'pending' (only pending can be edited)
        - Build dynamic UPDATE query with only provided fields
        - Update updated_at timestamp
        - Return updated order

      async def update_order_status(db, merchant_id, order_id, status, note) -> dict:
        - Verify order belongs to merchant
        - Validate status transition:
            pending   → assigned, cancelled
            assigned  → picked_up, cancelled
            picked_up → in_transit
            in_transit→ delivered
        - Invalid transitions return 400
        - UPDATE orders SET status = $new_status, updated_at = NOW()
        - INSERT into order_status_history
        - Return updated order with status_history

      async def cancel_order(db, merchant_id, order_id) -> dict:
        - Verify order belongs to merchant
        - Verify order is 'pending' (via DELETE endpoint)
        - Set status to 'cancelled'
        - Add status_history entry
        - Return success

  [ ] 3. Create endpoints
      File: backend/app/routers/orders.py (add to existing)

      GET /api/orders/{order_id}
        - Protected
        - Validate order_id is valid UUID
        - Call get_order_detail()
        - Return { data: OrderDetailResponse }

      PUT /api/orders/{order_id}
        - Protected
        - Only works when status = 'pending'
        - Validate input via OrderUpdate
        - Call update_order()
        - Return { data: order, message: "Order updated" }
        - 400 if order is not pending

      PATCH /api/orders/{order_id}/status
        - Protected
        - Validate via StatusUpdateRequest
        - Call update_order_status()
        - Return { data: order, message: "Status updated to {status}" }
        - 400 if invalid transition

      DELETE /api/orders/{order_id}
        - Protected
        - Only works when status = 'pending'
        - Call cancel_order()
        - Return { message: "Order cancelled" }
        - 400 if order is not pending


STATUS TRANSITION MAP
=====================
  Current      → Allowed Next
  ──────────────────────────────
  pending      → assigned, cancelled
  assigned     → picked_up, cancelled
  picked_up    → in_transit
  in_transit   → delivered
  delivered    → (terminal - no transitions)
  cancelled    → (terminal - no transitions)

  Store this as a dict in the service:
  VALID_TRANSITIONS = {
      'pending':    ['assigned', 'cancelled'],
      'assigned':   ['picked_up', 'cancelled'],
      'picked_up':  ['in_transit'],
      'in_transit': ['delivered'],
      'delivered':  [],
      'cancelled':  [],
  }
