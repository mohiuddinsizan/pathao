STATUS TRANSITION RULES
========================
  Each order can only move to certain next states:
    pending    → assigned, cancelled
    assigned   → picked_up, cancelled
    picked_up  → in_transit
    in_transit → delivered
    delivered  → (done)
    cancelled  → (done)

  The simulation UI shows buttons ONLY for allowed next states.

TASK 1 — Backend: PATCH /api/orders/{order_id}/status
=======================================================
  File: backend/app/routers/orders.py

  Add a new route at the bottom of this file.

  What it should do:
    1. Accept request body with "status" (string) and optional "note" (string).
       You can use `data: dict` like existing endpoints, or create a Pydantic
       model — either is fine.
    2. Fetch the order: WHERE order_id = $1 AND merchant_id = $2
       (use PTH string for order_id, like all other routes in this file)
       Return 404 if not found.
    3. Check if data["status"] is in the allowed next-states list (see table
       above). Return HTTP 400 with a helpful message if it is not allowed.
    4. Run two SQL statements:
         UPDATE orders SET status = $new_status, updated_at = NOW()
           WHERE order_id = $1
         INSERT INTO order_status_history (order_id, status, note)
           VALUES ($uuid_id, $status, $note)
       Note: order_status_history.order_id is the UUID id, not the PTH string.
       Fetch row["id"] first, then use that for the INSERT.
    5. Return the updated order row.

  Imports you may need:
    from typing import Optional  (if you use typed Pydantic model)
    The router, Depends, HTTPException, get_current_merchant, get_db are all
    already imported at the top of this file.
